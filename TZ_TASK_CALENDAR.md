# Техническое задание: приложение «Task Calendar»

## 1. Общая информация
- **Название:** Task Calendar
- **Тип:** Desktop-приложение (Python + Tkinter)
- **Цель:** управление задачами с привязкой к дате/времени и напоминаниями.
- **Целевая аудитория:** личное планирование (1 пользователь, локальное хранение).

## 2. Область работ
В рамках MVP реализовать:
1. Календарный выбор даты.
2. Список задач на выбранный период (день/неделя/месяц).
3. CRUD операций для задач.
4. Напоминания перед дедлайном.
5. Поиск и фильтрацию задач.
6. Локальное хранение данных в JSON.

## 3. Функциональные требования

### 3.1 Управление задачами
Пользователь может:
- Создавать задачу.
- Редактировать задачу.
- Удалять задачу.
- Отмечать задачу как выполненную/невыполненную.

#### Поля задачи
- `id` (string, UUID)
- `title` (string, обязательно, 1..120 символов)
- `description` (string, опционально, до 1000 символов)
- `date` (string, формат `YYYY-MM-DD`, обязательно)
- `time` (string, формат `HH:MM`, опционально)
- `completed` (boolean)
- `priority` (enum: `low|medium|high`, default `medium`)
- `remind_before` (integer, минут до события, default `30`)
- `created_at` (ISO datetime)
- `updated_at` (ISO datetime)

### 3.2 Календарь
- Отображение задач по датам.
- Выбор даты через календарный контрол.
- Подсветка дней, в которых есть задачи.
- Режимы просмотра:
  - День
  - Неделя
  - Месяц

### 3.3 Уведомления
- Напоминание о задаче за N минут до времени задачи.
- N задается в поле `remind_before`.
- Напоминание появляется один раз для конкретной задачи/времени.
- Для задач без `time` уведомления не создаются.

### 3.4 Фильтры, сортировка, поиск
- Фильтр по статусу: все / активные / выполненные.
- Фильтр по приоритету: все / low / medium / high.
- Фильтр по периоду: день / неделя / месяц.
- Поиск по названию задачи (`title`, частичное совпадение, регистронезависимо).
- Сортировка по умолчанию: `date ASC`, `time ASC`, `priority DESC`.

## 4. Нефункциональные требования
- **Производительность:** отклик UI < 150 мс для операций до 5000 задач.
- **Надежность:** сохранение данных при каждом изменении задачи.
- **Валидация:** строгая проверка форматов даты/времени.
- **Юзабилити:** интерфейс на русском языке, минимальное количество действий.
- **Совместимость:** Python 3.10+.

## 5. UI/UX требования

### 5.1 Основные экраны
1. **Главный экран**
   - Панель выбора даты/периода.
   - Календарь месяца с подсветкой дней с задачами.
   - Таблица задач.
2. **Форма задачи (создание/редактирование)**
   - Поля: название, описание, дата, время, приоритет, напоминание.
3. **Диалоги подтверждения**
   - Подтверждение удаления.
   - Сообщения об ошибках валидации.

### 5.2 Поведение выполненных задач
- Опция 1 (MVP): показывать в списке со статусом `completed`.
- Опция 2 (после MVP): отображение зачеркнутым текстом.

## 6. Хранение данных

### 6.1 Формат данных (JSON)
```json
{
  "tasks": [
    {
      "id": "3f9a4aab-c4ea-4f1a-9a38-45a1d9f4ec9d",
      "title": "Купить продукты",
      "description": "Молоко, хлеб",
      "date": "2026-03-25",
      "time": "18:00",
      "completed": false,
      "priority": "medium",
      "remind_before": 30,
      "created_at": "2026-03-25T10:30:00",
      "updated_at": "2026-03-25T10:30:00"
    }
  ]
}
```

### 6.2 Правила сохранения
- Файл хранится локально: `tasks.json`.
- Запись происходит после каждой CRUD-операции.
- При повреждении JSON приложение должно стартовать с пустым списком и показать предупреждение.

## 7. Архитектура (desktop MVP)
- `work.py` — UI + контроллер.
- Слой данных:
  - `load_tasks()`
  - `save_tasks()`
  - `validate_task()`
- Слой бизнес-логики:
  - фильтрация/поиск/сортировка
  - расчет диапазонов периодов
  - планировщик уведомлений

## 8. Бизнес-правила
1. Нельзя создать задачу без `title` и `date`.
2. `time` может быть пустым.
3. `remind_before` >= 0.
4. При редактировании `id` не меняется.
5. Выполненная задача не должна вызывать новые уведомления.

## 9. Ограничения и допущения
- Один локальный пользователь.
- Без облачной синхронизации в MVP.
- Без авторизации в MVP.
- Часовой пояс берется из системного времени ОС.

## 10. Критерии приемки (Acceptance Criteria)
1. Пользователь может создать, изменить, удалить и отметить задачу выполненной.
2. В режиме «день» отображаются только задачи выбранной даты.
3. В режиме «неделя/месяц» отображаются задачи в границах периода.
4. Поиск по названию и фильтры работают совместно.
5. Напоминание появляется в нужное время (с учетом `remind_before`) и только один раз.
6. После перезапуска приложения данные остаются в `tasks.json`.

## 11. План тестирования

### 11.1 Позитивные сценарии
- Создание задачи с датой без времени.
- Создание задачи с датой и временем.
- Редактирование названия/даты/времени.
- Удаление задачи с подтверждением.
- Переключение статуса completed.
- Поиск по части слова.
- Фильтрация по статусу и приоритету.

### 11.2 Негативные сценарии
- Неверный формат даты (`2026/03/25`).
- Неверный формат времени (`25:61`).
- Пустой заголовок.
- Отрицательное значение `remind_before`.

### 11.3 Регрессия
- Проверка сохранения после каждой операции.
- Проверка загрузки ранее сохраненных задач.
- Проверка отсутствия дублированных уведомлений.
